HTTP & Runtime

How Polonio bridges the CLI renderer with CGI environments and (future) dev-server mode.

CGI Auto-Detect

When polonio runs with no CLI arguments and GATEWAY_INTERFACE is present, main.cpp routes execution through handle_cgi_request(). The flow:

  1. build_cgi_context() (in runtime/cgi.cpp) collects environment variables, parses the request, and discovers the script via SCRIPT_FILENAME or PATH_TRANSLATED.
  2. The interpreter loads the template file, installs superglobals, and binds a ResponseContext so HTTP helpers become available.
  3. After rendering, ResponseContext::emit flushes headers (including Status: when non-200) followed by the template output.
CGI mode throws an error if it cannot find a script path; make sure your web server sets SCRIPT_FILENAME.

Superglobals

Inserted into the interpreter’s global environment right before template execution:

NameSource
_GETParsed query string (application/x-www-form-urlencoded) into scalars or arrays when a parameter appears multiple times.
_POSTFor POST requests with CONTENT_TYPE=application/x-www-form-urlencoded. The runtime reads exactly CONTENT_LENGTH bytes from stdin; anything beyond that is ignored.
_COOKIESemicolon-delimited HTTP_COOKIE header with percent-decoding.
_SERVERCommon CGI variables + every HTTP_* entry exposed by the web server.
Polonio treats each request separately. Superglobals are regenerated on every run, so mutating them only affects the current request.

Response Helpers

With an attached ResponseContext, templates can control headers and status codes directly.

  • http_status(code) / status(code): integer 100–599.
  • http_header(name, value) / header(...): add arbitrary headers. The helper prevents header injection by validating newline-free values.
  • http_content_type(value): sets Content-Type and removes previous ones so only one sticks.
  • redirect(target[, status]): writes the status (default 302) and a Location header.
<%
http_status(200)
http_content_type("text/html; charset=utf-8")
http_header("X-Polonio", "v0.1.0")
%>

URL Utilities

Use urlencode and urldecode (builtins) for query strings and cookie-safe encoding. They follow the same rules as _GET/_POST parsing: space becomes +, and percent-escapes are upper/lowercase tolerant.

<%
var q = urlencode("hola mundo")
echo "/search?q=" .. q
%>

Headers & Body Emission

ResponseContext guards output:

  • Headers accumulate until the first emit(). If you set multiple Content-Types, only the latest remains (add_header removes earlier ones).
  • If you never call http_content_type, emit() adds a default Content-Type: text/html.
  • Non-200 statuses appear as Status: NNN before the header block.
  • After emitting headers, further header calls raise “headers already sent”.
CGI stdin handling is strict: only `CONTENT_LENGTH` bytes are read, so be sure your server sets it correctly for POST bodies.

Dev Server (Stub)

polonio serve [--port N] [--root DIR] now binds to localhost, validates the requested root directory, logs the URL, and responds to the first HTTP connection with a fixed 200 OK text body. This scaffolding powers upcoming work listed in docs/FEATURE_QUEUE.md:

  • Static file routing rooted at the provided directory.
  • Template rendering with CGI-parity superglobals and POST handling.
  • Response helper support (http_status, redirect, etc.).
Use the stub mainly to verify CLI plumbing or watch for port conflicts; real static/template serving will land in later sub-milestones.
